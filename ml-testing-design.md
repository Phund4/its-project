## Шаг 1. Подготовка данных

Подготовка выполнена на основе источников из ЛР1 и стратегии валидации из ЛР2.

### 1.1. Загрузка и предобработка

- Источник данных: видеоконтур ИТС.
- Для экспериментов использованы:
  - `CCTV_accidents` для трека `accident` (классы `Accident` / `Non Accident`);
  - `video_ground_truth_eval` для трека `congestion`.
- Предобработка соответствует инференсу:
  - ресайз до входного размера чекпойнта;
  - преобразование в тензор и нормализация.

### 1.2. Разделение train/val/test

Так как обучение не выполняется, разделение используется для корректного протокола валидации:
- разбиение делается **по независимым единицам** (видео/сессии), а не по случайным кадрам одной записи (во избежание утечки);
- фактический отчётный прогон:
  - `accident` -> `CCTV_accidents/test`;
  - `congestion` -> фиксированный тестовый корпус `video_ground_truth_eval`.

### 1.3. Feature engineering

Feature engineering в классическом смысле **не применялся**, чтобы не менять входное пространство модели. Используются только преобразования, совместимые с инференсом.

### 1.4. Фиксация в коде

Предобработка и оценка зафиксированы в скриптах:
- `ml-experiments/scripts/run_pipeline.sh`
- `ml-experiments/scripts/evaluate_video_gt.py`
- `ml-experiments/inference_core.py`

## Шаг 2. Выбор и реализация baseline-модели

Поскольку обучение baseline запрещено условиями работы, baseline трактуется как **самая простая/лёгкая из готовых моделей**.

- Для трека `accident`: baseline = `baseline_cnn` (сравнение с `resnet18`).
- Для трека `congestion`: baseline = `tiny_cnn` (сравнение с `linear_resnet`).

Обучение в рамках отчёта не описывается; запускались только инференс-прогоны `best.pt` с вычислением метрик.

## Шаг 5. Анализ ошибок

### 5.1. Выявленные слабые места

- Для `accident` наблюдается компромисс между классами: `baseline_cnn` даёт высокий `recall_crash=1.0`, но полностью проваливает `normal` (`f1_normal=0.0`), а `resnet18` более сбалансирован по классам.
- У тяжёлых моделей (`resnet18`, `linear_resnet`) значительно хуже latency/fps, что создаёт риск срыва SLA при потоке камер.

### 5.2. Причины

- Набор `CCTV_accidents/test` ограничен по размеру (`100` изображений), поэтому метрики чувствительны к смещению распределения сцен.
- Возможен сдвиг домена между данными, на которых обучались чекпойнты, и текущим видео из контура.

### 5.3. Способы улучшения (без выполнения в этой ЛР)

- Расширить тестовый accident-набор кадрами с подтверждёнными инцидентами.
- Провести калибровку порогов/постобработки для снижения false positive.
- Добавить контроль по сегментам/камерам для диагностики доменного сдвига.

## Шаг 6. Выбор финальной модели

Финальный выбор для инференса:
- `accident` -> `resnet18`
- `congestion` -> `tiny_cnn`

Обоснование выбора:

1. **Качество по accident:** `resnet18` даёт более сбалансированное качество между `crash` и `normal` (лучший итоговый баланс классов), в отличие от `baseline_cnn`, который уходит в перекос на один класс.
2. **Качество по congestion:** `tiny_cnn` лучше по `MAE/RMSE` и остаётся значительно быстрее альтернативы.
3. **Компромисс сложность vs качество:** для accident выбран более тяжёлый, но более устойчивый по классам вариант; для congestion — лёгкая и точная модель.
4. **Соответствие нефункциональным требованиям:** обе выбранные модели укладываются в практически применимый latency-профиль для текущего контура.

Формат сохранения модели для инференса: `best.pt` (PyTorch checkpoint), пригодный для загрузки в `ml-serving`.
